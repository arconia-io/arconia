= OpenLLMetry Semantic Conventions

Spring AI provides built-in instrumentation for models, vector stores, and workflows based on the Micrometer APIs. Arconia lets you configure your Spring AI applications to export telemetry data adopting the https://github.com/traceloop/openllmetry[OpenLLMetry Semantic Conventions] instead of the default Micrometer conventions, enabling rich trace visualization in platforms like https://www.traceloop.com[Traceloop] and other OpenTelemetry-compatible backends.

[WARNING]
====
The https://github.com/traceloop/openllmetry[OpenLLMetry Semantic Conventions] are still experimental, and they might change in the future.
====

== Getting Started

Add the Arconia OpenLLMetry Semantic Conventions dependency to your project. The module auto-activates when present on the classpath.

[tabs]
======
Gradle::
+
[source,groovy]
----
dependencies {
  implementation 'io.arconia:arconia-openllmetry-semantic-conventions'
}
----

Maven::
+
[source,xml]
----
<dependency>
    <groupId>io.arconia</groupId>
    <artifactId>arconia-openllmetry-semantic-conventions</artifactId>
</dependency>
----
======

Your application also needs OpenTelemetry configured for exporting traces. The recommended approach is to use the xref:opentelemetry:index.adoc[Arconia OpenTelemetry Spring Boot Starter]:

[tabs]
======
Gradle::
+
[source,groovy]
----
dependencies {
  implementation 'io.arconia:arconia-spring-boot-starter-opentelemetry'
}
----

Maven::
+
[source,xml]
----
<dependency>
    <groupId>io.arconia</groupId>
    <artifactId>arconia-spring-boot-starter-opentelemetry</artifactId>
</dependency>
----
======

== Features

The module provides semantic conventions for the following observation types:

* Chat model spans with full parameter tracking (e.g., temperature, max tokens, penalties, stop sequences) using standard `gen_ai.*` attributes.
* Embedding model spans with dimension tracking.
* Tool execution spans with `traceloop.entity.input` and `traceloop.entity.output` attributes.
* Chat client observations with workflow tracking via `traceloop.span.kind` and conversation ID via `traceloop.association.properties`.
* Token usage metrics (`gen_ai.usage.input_tokens`, `gen_ai.usage.output_tokens`, `gen_ai.usage.total_tokens`).
* Input/output message content, captured as JSON in `traceloop.entity.input` and `traceloop.entity.output` span attributes.

Content capturing is enabled by default, matching the behavior of the official OpenLLMetry SDKs.

== Configuration Properties

.OpenLLMetry Configuration Properties
|===
|Property |Default |Description

|	`arconia.observations.conventions.openllmetry.trace-content`
|	`true`
|	Whether to include input and output content in traces (messages, tool arguments, tool results).

|	`arconia.observations.conventions.openllmetry.include-tool-definitions`
|	`true`
|	Whether to include tool definitions in traces.
|===

== Environment Variables

Arconia supports the https://www.traceloop.com/docs/openllmetry/configuration[OpenLLMetry/Traceloop environment variables] for configuring the OpenLLMetry semantic conventions. If both the environment variables and the Arconia configuration properties are set, the environment variables will take precedence.

.Supported Environment Variables
|===
|Environment Variable |Description

|	`TRACELOOP_TRACE_CONTENT`
|	Whether to capture trace content (`true` or `false`). Maps to `arconia.observations.conventions.openllmetry.trace-content`.
|===
